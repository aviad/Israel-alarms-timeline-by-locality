* What Was Learned: Cloudflare Python Workers Reality (March 2026)

** Handler API: two incompatible modes

There are two distinct Python Worker APIs. They are incompatible and have
different size limits:

| Mode         | Handler              | Size limit (gzip) | Packages            |
|--------------+----------------------+-------------------+---------------------|
| Old          | =on_fetch(req, env)= | 64 MiB upload     | none (broken)       |
| New (active) | =WorkerEntrypoint=   | 3 MiB free        | vendored via pywrangler |
|              | class + =fetch=      | 10 MiB paid       |                     |

The old =on_fetch= handler is fully deprecated as of wrangler 4.71.0.
Even with =compatibility_flags = ["python_workers"]=, the runtime no
longer registers =on_fetch= as a handler. Only =WorkerEntrypoint= works.

*** Key symptom
Importing =from workers import WorkerEntrypoint= switches wrangler into
the new mode with the 3/10 MiB gzip limit. Without that import, the old
mode is used (higher size limit) but the handler is never found.

** Package management: pywrangler, not wrangler

The new Python Workers use =pywrangler= (=uv run --with pywrangler pywrangler=):
- =pywrangler sync= installs packages from =pyproject.toml= dependencies
  into =python_modules/= using Pyodide-compatible wheels (WASM .so binaries)
- =pywrangler deploy= runs sync then proxies to wrangler
- =requirements.txt= and =cf-requirements.txt= are both rejected by wrangler
- The =workers= module (=WorkerEntrypoint=, =Response=, etc.) is a Cloudflare
  built-in, not installable via pip

*** python_modules/ artifacts to remove before deploy
=pywrangler sync= creates several artifacts that pollute the bundle:
- =.venv/= — local desktop Python venv (~200 MB), wrangler picks up all
  its .py files. Must be deleted before deploy.
- =.venv-workers/= — Pyodide dev venv (~167 MB), same problem.
- =python_modules/_virtualenv.py=, =python_modules/_virtualenv.pth= —
  virtualenv hook files; remove them (cause issues in Pyodide runtime)
- =python_modules/pyvenv.cfg=, =python_modules/.synced= — marker files;
  remove them

=.wranglerignore= does NOT reliably exclude these directories.
Delete them manually before deploying.

** CPU time: the fatal blocker

matplotlib in Pyodide WASM far exceeds the 30-second CPU limit.
The landing page (no rendering) works. The chart endpoint (matplotlib
rendering) hits error 1102 every time.

This is fundamental: Python WASM execution is ~10-50x slower than native.
A chart that renders in 1 second locally takes 30+ CPU-seconds in Pyodide.

Even pre-warming the worker (cron trigger) would not help because the
CPU limit applies per-request, not per cold-start.

** ZoneInfo / tzdata

=zoneinfo= requires the =tzdata= package in Pyodide
(=pyodide.loadPackage('tzdata')=). Without it:
  =ZoneInfoNotFoundError: No time zone found with key Asia/Jerusalem=
Fix: moved =ISRAEL_TZ = ZoneInfo(...)= out of =alarms_core.py= (which
the worker imports) into =alarms_graph.py= (CLI only). The worker does
not need timezone-aware datetimes.

** seaborn

Not available in Pyodide. =try/except ImportError= guard works fine.
Fallback: set spines manually with matplotlib.

** PIL / Pillow

matplotlib 3.8 imports PIL unconditionally at module level in:
- =matplotlib/colors.py=: =from PIL import Image= and =PngImagePlugin.PngInfo=
- =matplotlib/image.py=, =backends/backend_*.py=, =animation.py= etc.

PIL is only USED in =colors.py= for =Colormap._repr_png_()= (Jupyter
notebook colormap preview). Our rendering path never triggers this.

Solution: stub PIL with 4 minimal Python files (~20 lines total).
This avoids bundling PIL's WASM binary (~3 MB gzip).

* The Fix: Replace matplotlib with Pure SVG Rendering

matplotlib in WASM is fundamentally too slow. The correct solution is
to generate SVG directly from Python — no C extensions, no WASM, no
Pyodide overhead.

** Why this works

The chart layout is fixed and predictable:
- Fixed coordinate system: x = hour (0–24), y = one row per day
- Night shading: two rectangles (x=0–7 and x=21–24)
- Alert marks: vertical lines at exact time positions (lines style) or
  circles scaled by count (dots style)
- Y-axis labels: date strings positioned at fixed row centers
- X-axis labels: hour labels at 0, 3, 6, ... 24
- Total count circles on the right: split pie chart using SVG arcs
- Title and subtitle: positioned text elements

All of this is standard SVG. Python can generate it as a string.
No numpy required (basic arithmetic only).
No matplotlib required.
No PIL required.

** What changes

=alarms_core.py= currently:
- Imports =matplotlib.font_manager=, =matplotlib.path.Path=,
  =matplotlib.pyplot= at module level
- =render_chart()= uses matplotlib/seaborn throughout

New =alarms_core.py=:
- No matplotlib import at all
- =render_chart()= returns SVG bytes built from f-strings/string concat
- =fmt= parameter: only "svg" makes sense; PNG support dropped for worker
  (local CLI can still use matplotlib directly in =alarms_graph.py=)

=alarms_graph.py= keeps its own matplotlib import and =plot()= function
for local display. It calls =render_chart()= from =alarms_core= for file
output, OR we split:
- =alarms_core.render_chart()= → SVG bytes (worker-safe)
- =alarms_graph.py= keeps a local =render_png()= using matplotlib for
  the =--output .png= case, and calls =render_chart()= for SVG

No need for =pywrangler= or =pyproject.toml=.
Works on *free plan* (no matplotlib = no CPU issue).

*** SVG wedge (pie arc) for night/day total dot

The existing matplotlib code uses a custom =_make_wedge_marker()= with
a =Path=. In SVG this becomes a standard arc path:

#+begin_src python
def _svg_wedge(cx, cy, r, start_frac, end_frac, color):
    """SVG arc path for a pie wedge, fractions 0–1 clockwise from top."""
    import math
    t0 = math.pi / 2 - start_frac * 2 * math.pi  # start angle
    t1 = math.pi / 2 - end_frac * 2 * math.pi    # end angle
    x0, y0 = cx + r * math.cos(t0), cy - r * math.sin(t0)
    x1, y1 = cx + r * math.cos(t1), cy - r * math.sin(t1)
    large = 1 if (end_frac - start_frac) > 0.5 else 0
    return (f'<path d="M {cx},{cy} L {x0:.1f},{y0:.1f} '
            f'A {r},{r} 0 {large},1 {x1:.1f},{y1:.1f} Z" '
            f'fill="{color}"/>')
#+end_src

** Local CLI after SVG switch

Options for =alarms_graph.py=:
1. Keep using matplotlib's =render_chart()= locally by keeping a copy
   of the old plot() function in =alarms_graph.py= itself. Call
   =alarms_core.render_chart()= for file output.
2. Or: =alarms_graph.py= imports =render_chart= from =alarms_core=
   (gets SVG), writes to file, then uses =subprocess= or
   =webbrowser.open()= to display. Simpler, SVG viewers are everywhere.
3. Or: keep =alarms_core.render_chart()= returning SVG but add a
   separate =alarms_core.render_chart_png()= that uses matplotlib,
   and have =alarms_graph.py= call that for display.

Recommended: option 2 is cleanest. The CLI outputs SVG by default.
=plt.show()= is replaced by writing SVG to a temp file and opening it.
